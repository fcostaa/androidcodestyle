Orientações de Estilo

A fim de manter o código consistente, utilize as seguintes convenções. De agora em diante "bom" e "mau" são usados ​​para atribuir coisas que fariam o estilo de código corresponder, ou não. Isto não é um julgamento sobre suas habilidades de codificação, porém é mais uma chamada de estilo e aparência. Por favor, tente seguir estas orientações para garantir a clareza.
Você também pode querer ler o Convenção de Código para Linguagem Java também, no entanto, as diretrizes abaixo prevalecem para o código Java, se houver um conflito.

Nomes

Utilize Nomes Significativos
Escolha nomes que sugerem o seu propósito. Bons nomes ajudam o desenvolvedor a entender o problema que você está resolvendo.
Bom: 
int count;

List newsList;
Mau: 
int a;

List someList;
Nome de Variáveis
Comece com uma letra minúscula e use letras maiúsculas como separador. Não utilizar underscore ('_').
Bom: 
int myVar;

int theInt = 1;

String someString = "Hello";
Mau: 
int my_var;

int theint = 1;

String some_string = "Hello";
Nome de Constantes
Use todas as letras maiúsculas e utilize underscore ('_') como separador.
Bom: 
final int MY_CONST = 1;
Mau: 
final int myConst = 1;
Nom de Métodos
Comece com uma letra minúscula e use letras maiúsculas como separador. Não utilize underscore ('_').
Bom: 
int myMethod();

void fooAndBar();
Mau: 
int My_Method();

void foo_and_bar();
Nome de Classes
Comece com uma letra maiúscula e utileze letras maiúsculas como separador. Não utilize underscore ('_').
Bom: 
public class MyClassName

class Example
Mau: 
public class My_class_Name

class example_

Chaves

Convenção
Uma questão fervoroso de grande debate nos círculos de programação é a colocação de chaves. O estilo Java é colocar a chave inicial na mesma linha que a palavra-chave e a última chave em sua própria linha, mas alinhado com a palavra-chave.
Bom: 
/**
 * Braces
 */
interface Empty {
}

enum MyEnum {
    UNDEFINED(0) {
	void foo() {
	}
    }
}

@interface SomeAnnotationType {
}

class Example {
    SomeClass fField = new SomeClass() {
    };
    int[] myArray = { 1, 2, 3, 4, 5, 6 };
    int[] emptyArray = new int[] {};

    Example() {
    }

    void bar(int p) {
	for (int i = 0; i < 10; i++) {
	}
	switch (p) {
	case 0:
	    fField.set(0);
	    break;
	case 1: {
	    break;
	}
	default:
	    fField.reset();
	}
    }
}
Mau:

Quando Chaves são Necessários
Todas as declarações if, while e do devem ter chaves ou estar em uma única linha. Isso ajuda a garantir que alguém ao adicionar uma linha de código não esqueça de adicionar chaves.
Bom: 
/**
 * If...else
 */
class Example {
    void bar() {
	do {
	} while (true);
	try {
	} catch (Exception e) {
	} finally {
	}
    }

    void foo2() {
	if (true) { return; }
	if (true) {
	    return;
	} else if (false) {
	    return;
	} else {
	    return;
	}
    }

    void foo(int state) {
	if (true) return;
	if (true) return;
	else if (false) return;
	else return;
    }
}
Mau:

Espaços em branco

Sem Carácter de Tabulação
Não tenha quaisquer caracteres de tabulação em seu código fonte. É difícil ou impossível de ler o código fonte se as suas configurações de tabulação forem diferentes dos demais desenvolvedores. A maneira mais fácil de remover a tabulação é definir em seu editor de texto que substituia o número correto de espaços para um caractere de tabulação.
Comprimento da Linha
Limite o seu comprimento da linha a 80 caracteres, já que linhas mais longas podem causar problemas com muitos editores de texto e outras ferramentas.
Bom: 
/**
 * Declaration
 */
class Example {
    public final synchronized java.lang.String aMethodWithALongName() {
    }
}

/**
 * Parameters
 */
class Example {
    void foo(int arg1, int arg2, int arg3, int arg4, int arg5, int arg6) {
    }
}

/**
 * 'throws' clause
 */
class Example {
    int foo() throws FirstException, SecondException, ThirdException {
	return Other.doSomething();
    }
}

/**
 * Declaration
 */
class Example {
    public final synchronized java.lang.String aMethodWithALongLongLongLongName() {
    }

/**
 * Parameters

class Example {

 * 'throws' clause
 */

	return Other.doSomething();
    }

Espaçamento ao redor de Operadores
Sempre coloque espaços antes e depois de operadores binários. Isso melhora a legibilidade de expressões.
Bom:

boolean b= i > 0 && i < 10 || i == 50;
Mau: 
int c =-a*b-d;

boolean b= i>0&&i<10||i==50;

if-else-if-else
Sempre alinhe as declarações if com as chaves para a sua declaração else associada. Especificamente, coloque a chave inicial na mesma linha que a palavra-chave e a linha de fuga na mesma linha que a próxima instrução.
Bom: 
if (condition) {               
...
} else if (condition) {    
...
} else {                      
...
}
Mau: 
if (condition) 
{               
...
} 
else if (condition) 
{    
...
} 
else 
{                      
...
}

Adições ao Javadoc

Sem números mágicos
Um número mágico é um literal numérico que não é definido como uma constante. É mágico porque ninguém tem a menor idéia do que isso significa que depois de 3 meses, incluindo o autor. De uso generalizado, -1, 0, 1 e 2, não são considerados números mágicos. Sempre que você atribuir um número a uma variável, use uma constante em seu lugar. Em Java, você declara constantes, que são variáveis ​​que não podem mudar, usando a palavra-chave final. Você pode usar constantes locais dentro de métodos ou membros constantes declaradas fora de qualquer método. Normalmente, você declarar constantes membro como uma variável membro public static final.
Bom: 
public static final int MY_CONSTANT = 10;
...
calculate(MY_CONSTANT);
Mau: 
calculate(10);
Tamanho de Métodos
Os métodos devem ter não mais do que 150 linhas. Se um método torna-se muito longo, é difícil de entender. Em vez disso, você deve criar métodos menores que se concentram em tarefas individuais.
Caractere de tabulação
Não possua quaisquer caracteres de tabulação em seu código fonte. É difícil ou impossível de ler o código fonte se as suas configurações de tabulação são de diferentes autores.

Comentários

Comentários são notas explicativas para os seres humanos lerem um programa. Com boas opções de nome, os comentários podem ser mínimos em um programa. Os únicos comentários requeridos são blocos de comentários pouco antes da declaração de classe (após as instruções de importação) e antes de cada declaração de método. Além de blocos de comentários, uma outra vez para adicionar comentários é quando o código é incomum ou obscuro. Quando algo é importante e não óbvio, merece um comentário.

Comentários de bloco
O Javadoc é um programa que analisa as declarações e comentários da documentação do seu código para produzir um conjunto de páginas HTML. Estas páginas descrevem seu código para outros programadores. Para um exemplo da documentação produzida, consulte a documentação da API do Java. Alguns Javadocs são derivado de blocos de comentários especialmente formatados, que você pode criar da seguinte forma:
  • Recue a primeira linha para alinhar-se com o código abaixo do comentário.
  • Comece o comentário com o símbolo de começar o comentário (/**), seguido de um retorno.
  • Comece as linhas subseqüentes com um asterisco *. Recue os asteriscos com um espaço adicional para alinhar os asteriscos. Separe o asterisco do texto descritivo ou a tag que seguir.
  • Adicione uma descrição a classe ou ao método.
  • Insira uma linha de comentário em branco entre a descrição e a lista de tags, como no exemplo.
  • Insera linhas em branco adicionais para criar várias tags.
  • A última linha começa com o símbolo de final de comentário (*/) recuado para a linha de asteriscos e seguido por um retorno. Note que o símbolo de final de comentário contém apenas um único asterisco. 
    /**
 * The main method for the HelloWorld program.
 *
 * @param args Not used
 */
  • Para mais informações sobre tags, veja o Javadoc .
Bloco de comentário de arquivo
Cada arquivo de código fonte (*.java) deve ter um bloco de comentário Javadoc, pouco antes da declaração da classe que contém o nome do autor, nome do arquivo e finalidade do arquivo. Uma ou duas linhas é geralmente suficiente para explicar o propósito.
Bom: 
import javax.swing.*;

/**
 * HelloWorld.java
 * Purpose: Prints a message to the screen.
 *
 * @author Jane User
 * @version 1.0 8/20/03
 */
public class HellowWorld {
Mau: 
import javax.swing.*;

/**
 * HelloWorld.java
 */

Bloco de comentário de método
Cada método deve ter um bloco de comentário Javadoc antes do método.
Bom: 
/**
 * Read a line of text from the shell console.
 *
 * @return A String input by the user.
 */
public String readLine();

/**
 * The following is some sample code which illustrates source formatting
 * within javadoc comments:
 * 
 * 
 * public class Example {
 *     final int a = 1;
 *     final boolean b = true;
 * }
 * 

 * 
 * Descriptions of parameters and return values are best appended at end of
 * the javadoc comment.
 * 
 * @param a
 *            The first parameter. For an optimum result, this should be an
 *            odd number between 0 and 100.
 * @param b
 *            The second parameter.
 * @return The result of the foo operation, usually within 0 and 1000.
 */
int foo(int a, int b);
Mau: 
public String readLine();

int foo(int a, int b);
